## 2. MVP 추가 축소 — "한 사이클 완전 동작" 정의

김지헌이 지난 회의의 문제의식("MVP 기능이 산발적으로 커진다")을 다시 강조:

- 지금 상태로는 **동작하지 않는 화면만 계속 만들고 있는 꼴**. 작더라도 **풀세트로 완전히 동작하는 기능 하나**를 먼저 만들고, 거기서부터 한 기능씩 확장해야 함.
- Codex에 "MVP 범위가 너무 넓다. 아주 작게 완전히 동작하는 기능을 먼저 만들고 넓혀가자"고 프롬프트.
- Codex 제안은 한 발 더 나아가 **로그인·DB까지 보류**하고 "혼자 쓰는 AI 요정 1회 생성"부터 만들자는 것이었으나 — 팀은 **로그인·DB·AI를 포함해야 진짜 '한 사이클'이 완성**된다고 판단해 포함하기로.

**확정된 최소 기능 한 사이클 (오늘의 목표 플로우)**:

1. 로그인 (Google)
2. 취향 카드 3개 이상 선택
3. 캐릭터 생성 → **DB에 저장**
4. "오늘의 우정주 받기" 클릭 → **AI가 사용자 캐릭터 기반으로 종목/테마 1개 생성**
5. 결과를 DB에 저장
6. 새로고침·재방문 후에도 오늘의 결과를 다시 볼 수 있음

> "쓸모없어 보여도 일단 기능적으로는 풀세트 하나가 만들어지는 것" — 이걸 베이스로 다음 기능을 붙여나간다.

작업 순서는 기존에 구현된 취향 카드를 그대로 살리고, 그 앞에 로그인을 붙이는 식으로. 친구 초대 등은 이번엔 만들지 않음(로그인된 사용자 단위가 잡히면 이후 확장).

## 3. 개발 환경 실시간 세팅

비개발자 팀원(덕우·태훈)에게 화면을 공유하며 환경 설정을 함께 진행. 김지헌은 "이런 환경 설정이 항상 제일 오래 걸리고 재미없는 작업"이라며, **로컬 환경을 생략하고 클라우드(Vercel 프리뷰)에서 직접** 작업하기로 한 이유도 설명(각자 로컬에 `.env`를 세팅하는 복잡도를 피하고, 민감 정보를 로컬에 두지 않기 위함).

### 3-1. 데이터베이스 — Neon (싱가포르 리전)

- 원래 쓰던 Supabase는 **무료 플랜이 프로젝트 2개 제한**이라 이미 가득 참 → 이번엔 **Neon** 사용(무료 0.5GB, 당장 충분).
- **리전 선택의 중요성**: DB 서버의 물리적 위치가 응답 속도를 좌우. 한국 리전은 없고 **가장 가까운 싱가포르** 선택(미국 리전은 왕복 지연이 큼). "클라우드 쓰다 보면 이런 리전·서버 위치 설정이 계속 나온다."
- 리소스 이름을 정해 DB 프로젝트 생성 → **프로비저닝 완료**. 생성 시 한 번만 보여주는 시크릿(`show secret`)은 노출되면 남이 우리 권한으로 DB를 조작할 수 있으니 즉시 복사·보관.

### 3-2. 환경변수 (Vercel)

- 시스템 구동 시 사용하는 변수 = 환경변수. Vercel 프로젝트(secondwind) 설정 → Environment Variables 에 등록.
- 이번에 등록한 값: **`DATABASE_URL`**, **`AUTH_SECRET`**, **`GOOGLE_CLIENT_ID`**, **`GOOGLE_CLIENT_SECRET`**.
- Gemini API 키는 **기존 secondwind 트래블에서 쓰던 키를 그대로 재사용**.

### 3-3. 인증 — Google OAuth (Google Cloud)

- 로그인은 **Google 인증** 사용. 김지헌이 공통 용도로 만들어 둔 Google Cloud 프로젝트(아쿠시랩스)에서 OAuth 클라이언트 설정 — "이 설정이 처음엔 제일 어려웠다."
- **테스트 계정 3명 등록**: 웹 게시(공개) 전까지는 등록된 테스트 계정(지헌·태훈·덕우의 Gmail)으로만 로그인 가능.
- **승인된 리디렉션 URI(redirect URI)** 추가 — 프리뷰에서 로그인을 테스트하려면 해당 프리뷰 URL도 추가해야 하는데, Vercel 프리뷰 URL은 PR마다 바뀔 수 있다는 점도 언급.
- Codex에 넘겨야 하는 건 **시크릿 값이 아니라 "체크 결과(존재 여부)"** 라는 점을 구분해서 진행.

## 4. 보안·키 관리 학습 (비개발자 대상)

이번 세션의 가장 큰 학습 주제. 김지헌이 환경 설정을 진행하며 반복 강조:

- **`.env`는 절대 Git에 올라가면 안 됨** → 그래서 `.gitignore`에 `.env`가 들어가 있음. 덕우가 이 흐름을 정확히 짚어냄.
- **AUTH_SECRET 생성**: 앱끼리 통신할 때 데이터를 서명·암호화하는 "앱 자체의 패스워드". 뚫리면 우리 서비스 통신 내용을 제3자가 캡처해 열어볼 수 있음. `base64` 인코딩 + 32바이트 난수로 임의 문자열 생성(`openssl rand -base64 32` 류). Google이 주는 값과 별개로 **우리 앱이 직접 만드는 비밀 키**.
- **환경변수 노출 위험 — 터미널 위치**: Codex(터미널 에이전트) 안에서 `!` 로 명령을 실행하면 그 출력이 **Codex/OpenAI 서버로 전송되어 학습될 수 있음**. 그래서 시크릿 생성·grep 같은 작업은 **Codex 밖의 기본 터미널에서 실행**해야 함. (이미 한 번 노출되면 에이전트가 "노출됐으니 키를 폐기하라"고 경고하기도 함.) 같은 맥락에서 "지금 이 Zoom도 우리 회의를 학습할 수 있다"는 농담 섞인 경계.
- **키 관리가 핵심**: 키는 한 번 만들 때만 보여주고 다시는 안 보여주는 경우가 많음 → 처음 보일 때 복사·보관 필수. 키 관리 체계가 무너지면 개인의 모든 게 뚫릴 수 있음.

## 5. 코드 구현 + 배포 확인

### 5-1. 구현 시작 (브랜치/PR 규약)

- 환경(DB·앱·인증)이 모두 준비된 뒤 Codex가 코드 작성 시작. "코드를 짜려면 DB·앱 정보가 먼저 필요"하므로 설정을 직렬로 먼저 보여준 것(평소엔 시간을 아끼려 코드부터 짜고 설정을 나중에 넣기도 한다는 점도 솔직하게 공유).
- 새 기능 작업 시 **main에서 작업 금지 → 별도 브랜치 생성 → PR 기반** 규칙은 `AGENTS.md`(=agent.md)에 정의되어 있고, Codex가 이 규약대로 자동으로 브랜치를 따고 진행.
- 도구 메모: Codex Desktop을 지운 상태라 **터미널(CLI)로 접속**해서 작업. `resume` 로 직전 대화를 이어받고, 모르는 건 **두 번째 터미널을 따로 띄워** 기존 작업을 건드리지 않고 질문. gstack(`/gstack`)은 요즘 Codex 등 에이전트가 그 수준 기능을 기본 내장해서 **굳이 안 써도 된다**는 판단도 공유.

### 5-2. 배포 결과 확인

회의 종료 직전 **프리뷰 배포 완료**. 화면 클릭으로 동작 확인:

- ✅ **Google 로그인** 동작 (김재환 계정으로 로그인)
- ✅ **취향 카드 선택** 동작 (의도적으로 한쪽으로 치우친 카드 선택 — "스토리가 끌린다", "유행 전에 먼저 발견하고 싶다" 등)
- ✅ **프로필/결과 DB 저장 + 세션 유지**: 로그아웃 후 재로그인·완전 새로고침해도 **선택한 취향과 결과가 그대로 유지**됨 → DB 저장 정상 확인
- ❌ **AI 우정주 추천 미동작**: "오늘의 우정주"가 느리게 돌다 제대로 생성 안 됨. **Gemini 연결이 제대로 안 된 것**으로 추정 — Gemini가 유료로 전환된 뒤 설정이 매번 까다롭고, 이번엔 연결이 안 잡힌 듯. 김지헌이 따로 디버깅하기로.

> "목표(한 사이클)는 일단 달성. AI 부분은 내가 따로 보겠다."

## 6. 잡담 — 건설업 디지털화·AI 자동화·자녀 교육

본 작업과 직접 관련은 없지만 기록.

- **건설업 디지털화 (남덕우)**: 본사 근무가 길어 현장으로 나갈 예정. 현장은 협력사 정보를 **수작업으로 엑셀에 입력**하는 게 여전히 중심 — 휴먼 에러·검증 불가 문제. 협력사 엑셀을 던지면 자동 취합·작성해주는 도구를 만들고 싶어 함. 더 나아가 **데이터별 표준 코드를 정의·관리하는 회사**(예: 일 출력 인원=A, 일 타설량=B …)를 차려, 그 표준 코드를 각 건설사에 배포하면 정보를 자동으로 뽑아낼 수 있다는 사업 구상. "회사 안에서가 아니라 개인 사업으로 해야 한다." (개인 노트북 필요 — 회사 노트북은 보안 정책으로 터미널·Claude/Codex 차단, Copilot만 허용)
- **AI 자동화 단계론 (김태훈)**: 사람들이 시연 영상의 레벨 4~5만 보고 "대박, 우리도 만들자" 하지만, **레벨 1부터 쌓아야** 함. 1단계(현장 인원 입출입을 엑셀에 기입·리포트하는 규칙 정의)처럼 괴롭고 사소해 보이는 과정을 쌓는 동안, 대부분의 회사가 "그동안 시스템 없이 사람이 멘땅에 헤딩했다"는 현실을 뼈저리게 마주하게 됨. AI에 무엇을 **입력**하고 무엇을 **출력**할지가 정의돼야 쓸모 있는 결과가 나옴(막연히 "현장 자동화" 맡기면 쓸모없는 걸 만듦).
- **ERP/SAP 류의 위기**: 갇힌(경직된) 사내 시스템은 포맷이 조금만 바뀌어도 못 쓰고 결국 사장됨. 표준화만 돼 있으면 AI로 언제든 원하는 형태를 뽑을 수 있어, 기존 ERP 솔루션 회사들이 AI로 빠르게 위협받을 수 있다는 관측. 대기업은 과감하지 못해 못 할 뿐.
- **AI 시대 자녀 교육 (김태훈)**: 초등 고학년부터 바이브 코딩 등 AI를 접하게 하면 어떨까. 입시만으로 미래가 안 그려지는 상황에서, AI에 일찍 노출시키는 학원도 이미 생기는 중. 2040년 수능 폐지·탈대학 담론 맥락에서 자녀의 일자리·교육을 AI 관점으로 바라보게 됨.
- **미래 전망**: "2년만 지나도 AI는 엑셀 같은 존재 — 당연히 써야 하고 못 하면 이상한 것". 그리고 결국 **개인이 AI를 잘 쓰는 단계를 넘어, 팀·조직 차원의 AI 협업 워크플로우(문화·환경·프로세스)를 설계할 수 있는 사람이 승자**가 된다 — 김지헌이 자기 팀에서 시도 중인 방향이기도 함.

## 주요 결정사항

- **MVP 추가 축소**: 로그인 → 취향 카드 3개 선택 → 캐릭터 생성 → AI 우정주 추천 → 결과 저장·재방문 확인까지를 **"완전히 동작하는 최소 한 사이클"** 로 정의. (AI가 로그인·DB 보류를 제안했으나, 풀 사이클 완성을 위해 포함하기로)
- **데이터베이스**: **Neon DB**, **싱가포르 리전**(한국 최근접). (기존 Supabase는 무료 프로젝트 한도 초과)
- **인증**: **Google OAuth**, 테스트 계정 3명 등록(웹 게시 전까지 테스트 계정만 접근).
- **개발 방식**: **로컬 환경 생략, Vercel 프리뷰(클라우드)에서 직접 작업** — 환경변수 세팅 복잡도 감소 + 민감정보 로컬 비저장.
- **LLM**: Gemini API 키는 기존 트래블 키 재사용.

## 완료된 작업

- Neon DB 프로젝트 생성 및 프로비저닝 완료 (싱가포르)
- Google Cloud OAuth 클라이언트 설정 + 테스트 계정 3명 등록 + 리디렉션 URI 추가
- 환경변수 등록: `DATABASE_URL`, `AUTH_SECRET`, `GOOGLE_CLIENT_ID`, `GOOGLE_CLIENT_SECRET`
- 로그인 + 취향 카드 선택 기능 구현·배포 완료
- 프로필/결과 DB 저장 및 세션 유지(재로그인·새로고침 후 유지) 확인

## 미완료 / 보류 사항

- **Gemini API 연결 실패**: AI 우정주 추천이 동작하지 않음 → **API 키·설정 재확인 및 디버깅 필요**. (Gemini 유료 전환 후 설정이 매번 까다로움)
- **취향 카드 기반 AI 추천 로직 구체화** 필요.
- **추천 결과 형식·데이터 구조 검토** 필요 — 스탯/결과가 **휴리스틱**인지 **AI 판단**인지 기준 정리.
- **로컬 환경 정리**: 시간이 없어 이번엔 로컬 세팅을 생략했고, 추후 키·설정을 다시 정리(기록)할 예정.
