## 1. 도구 사용 — Codex Desktop App 직접 운영해보기

지난 세션까지는 김지헌이 직접 Codex를 운영했지만, 이번에는 김태훈이 본인 PC에서 직접 Codex Desktop App을 띄워 작업을 리딩했다. 비개발자가 돌아가면서 손에 익히기 위함.

도구 옵션 비교:

- **VS Code + Codex 플러그인**: IDE 안에서 사용
- **Codex Desktop App**: 자체 데스크탑 앱 (이번 회의에서 사용). VS Code 거의 동등하게 동작.
- **Codex CLI**: 커맨드 라인 기반. 어렵지 않고 자연어 입력은 동일.

Codex Desktop App 사용 노하우:

- `@` 기호: 프로젝트 내 문서를 직접 참조. 파일명을 일부만 쳐도 자동완성됨.
  - 예: `@2026-05-09-friendship-stock-mvp.md @2026-05-10-finz-naming.md 이걸로 친구들과 아이디어를 모았어, 새 서비스 이름은 FINZ로 정했어`
- `!` 기호: 터미널 명령 실행 (단, Desktop App에서는 일부 제약이 있었음 — CLI 환경과 차이 있음).
- 자연어로 git 작업 지시 가능: "메인 브랜치로 체크아웃하고 최신화해줘" → 알아서 `git pull` 수행.
- 회의에서는 작업 전 `main` 브랜치 최신화 → 새 feature 브랜치 생성 → MVP 계획서 자동 생성 → 1단계 구현 순서로 진행.

기타 도구 관련 잡담:

- Google Antigravity가 최근 대폭 업데이트됨. Gemini 진영 코딩 에이전트.
- Gemini Flash Lite는 무료 한도에서 토큰이 너무 빨리 소모돼 실사용이 어려움.
- 결국 Claude가 시장을 장악할 것이라는 의견.

## 2. MVP 1 계획서 자동 생성 및 검토

Codex가 지난 회의 문서(`docs/ideation/2026-05-09-friendship-stock-mvp.md`, `docs/meetings/2026-05-10-finz-naming.md`)를 읽고 MVP 1 계획서와 최소 데이터 모델을 자동 생성했다.

자동 생성된 MVP 1 핵심 흐름:

1. 취향 카드 선택 시작 화면
2. 개인 투자 캐릭터 소환
3. 친구 그룹 생성 진입
4. (확장) 오늘의 우정주 제안
5. (확장) 투자 레이드 — 역할 기반 대화

제품 원칙 재확인:

- 증권 앱이 아니라 게임/대화 앱에 가까움.
- 종목 정답성보다 대화 가능성을 우선.
- 실제 금융, 돈, 자산은 다루지 않음.

검토 의견:

- MVP치고 화면이 다소 많다는 의견 (보통 이렇게 많이 만들지 않음).
- 1번(취향 카드)·2번(캐릭터 소환)·3번(그룹)은 가장 기본적인 기능, "없으면 안 돌아가는" 토대.
- 4번 우정주, 5번 레이드는 그 위에 얹는 **콘텐츠**에 해당. 카카오톡 위에 선물하기/투표/사다리타기가 붙는 구조와 비슷.
- 다만 레이드의 "보스" 같은 게임적 표현이 대화방 톤과 어울릴지에 대한 어색함이 남음. 이 부분의 지속 가능성에 대한 확신은 아직 부족.

## 3. 최소 데이터 모델 — 4개 엔티티 구조

Codex가 제안한 최소 데이터 모델을 함께 읽으며 비개발자 팀원에게 자료 구조를 설명.

비유: **구글 시트 = 테이블**, 각 시트의 열이 필드, 행이 실제 데이터.

| 엔티티 | 필드 |
|---|---|
| FINZ Taste Card | `id`, `label`, `tags` (배열 — 여러 태그 콤마 형태로 저장) |
| FINZ Character Card | `classId`, `className`, `level`, `title`, `summary`, `stats` (객체 — attack/defense 등 중첩 구조), `weakness`, `tips` |
| FINZ Member | `id`, `name`, `characterId`, `createdAt` |
| FINZ Group | `id`, `name`, `members` (배열), `createdAt` |

설명한 개념:

- `id`는 시스템이 자동 발번하는 고유키. 사람이 직접 신경 쓸 필요 없음.
- 사람의 식별자는 보통 `name`이지만, 컴퓨터는 이름 중복을 가정하므로 절대 `name`을 식별키로 쓰지 않음. `id`가 주민등록번호와 비슷한 역할.
- 배열(`[]`)이 들어가는 필드는 여러 값을 한 번에 담을 수 있다는 의미. 예: `FINZ Group.members`에 `[김지헌, 김태훈, 남덕우]` 같은 식으로 멤버 id들이 들어감.

## 4. MVP 1 구현 결과 — 첫 화면 확인

Codex가 작업을 마치고 `localhost:3000`에 개발 서버를 띄움. 브라우저 개발자 도구의 모바일 시뮬레이션(iPhone Pro 등 쿠폰 설정)으로 핸드폰 화면처럼 렌더링해서 확인.

직접 클릭해본 결과:

- 취향 카드 3장 선택 → 캐릭터 소환 버튼 → "미술 기술 딜러" 캐릭터 생성 확인.
- 캐릭터에는 이름, 레벨, 키워드, 한 줄 설명, 스탯, 약점, 친구 공유용 한 줄, 다음 레이드 역할이 함께 표시됨.
- 일단 동작은 함. AI는 연결되지 않은 상태이고 화면만 먼저 보여주는 단계.

디자인 관련 이슈:

- 현재 디자인이 SecondWind의 디자인 시스템을 그대로 따라와서 FINZ 컨셉과 톤이 잘 안 맞음.
- SecondWind 디자인 시스템이 `app/` 전반에 깔려 있어서 별도로 분리하지 않으면 FINZ만 다른 톤을 쓰기 어려움.
- 디자인 시스템 교체는 일단 보류. 추후 FINZ에 어울리는 참고 사이트를 모아 한 번에 교체할 예정.

디자인 참고 사례 수집 방법:

- ChatGPT/Codex에 프로젝트 소개와 MVP 기획서 주소를 던지고 어울리는 디자인 시스템 추천 받기.
- 평소 보다가 "기세 있다", "FINZ랑 잘 어울려 보인다" 싶은 사이트가 있으면 URL을 모아두기.
- Figma의 디자인 시스템이나 Claude Design이 음악적·시각적 톤을 잘 끌어옴 — Claude Design 결과물을 Claude Code로 이식하는 방식도 가능.

## 5. 로그인 제외 + 로컬 스토리지 결정

Codex 계획서가 의도적으로 로그인을 MVP 1에서 제외했다.

- MVP 1은 **로컬 스토리지 기반 개인 환경**에서만 동작. 새로고침해도 데이터는 유지되지만, 사용자 간 연결은 아직 안 됨.
- 즉, MVP 1은 "각자 자기 캐릭터를 뽑아보는" 단계까지만 작동. 친구와 실제로 연결되는 그룹/대화 기능은 다음 MVP에서.
- 로그인 + 다중 사용자 연결은 추후 단계로 미루기로 합의.

## 6. 기술 학습 — npm, 모노레포, 스킬

비개발자 팀원의 질문에 답하며 다음 개념을 짧게 설명.

**npm (Node Package Manager)**:

- 자바스크립트(Node.js) 기반 패키지 매니저.
- 비유: "자바스크립트 프로그램들의 앱스토어". 필요한 라이브러리를 `npm install`로 받아 씀.
- 비슷한 매니저: macOS의 Homebrew 등. AI가 다루기에 최적화된 도구 설치 방식.

**모노레포(Monorepo)**:

- 하나의 저장소(`/app` 폴더)에 여러 프로젝트를 동시에 띄울 수 있는 구조.
- 매번 프로젝트 세팅을 새로 하지 않아도 되게끔 김지헌이 secondwind를 모노레포로 구성해둠.

**스킬(Skill)**:

- 자주 쓰는 프롬프트와 명령 집합을 묶어놓은 것.
- 위치: `.agents/` 폴더 안에 `feature/SKILL.md` 같은 형태로 존재.
- gstack 스킬은 Codex/Claude 양쪽에서 모두 호출할 수 있도록 `bootstrap.sh`로 설치 경로를 맞춰둠.
- 강력하지만 토큰 소모가 크니 의식적으로 사용.

**브랜치 전략**:

- `main` 브랜치에는 직접 커밋 금지 (규칙으로 못 박아둠).
- 새 작업은 feature 브랜치를 따고, 완료되면 MR(Merge Request)을 만들어 리뷰 후 머지.

## 주요 결정사항

- **MVP 1 범위 확정**: 취향 카드 선택 → 개인투자 캐릭터 소환 → 그룹 생성 진입까지 구현.
- **로그인 제외**: MVP 1은 로컬 스토리지 기반 개인 환경에서만 동작.
- **데이터 저장 방식**: 로컬 스토리지 사용. 새로고침 시에도 데이터 유지.
- **디자인 시스템**: 현재는 SecondWind 디자인 시스템을 따르되, 추후 FINZ 전용 디자인으로 교체.
- **모노레포 유지**: `/app`에 모든 소스코드, `/docs`에 문서 관리.
- **MVP 화면 수 인정**: MVP치고 화면이 많지만, 1~3번은 기본기능이고 4~5번은 콘텐츠 레이어로 구분.
