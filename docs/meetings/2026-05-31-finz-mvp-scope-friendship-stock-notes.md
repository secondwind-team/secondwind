## 3. 구현 진행 방식에 대한 의문

김지헌이 지금 전개 방식이 일반적인 MVP와 다르다는 점을 지적:

- 보통 MVP는 **작게 완전히 동작하는 기능 하나**를 만들고 거기에 하나씩 붙여나가는 방식.
- 지금 Codex는 **동작하지 않는 화면 여러 개를 먼저 쫙 펼쳐놓고** 그 다음에 구현하려는 특이한 방식.
- 그룹 생성·초대, 우정주 추천 등은 백엔드(DB)가 있어야 실제 동작하는데 현재는 백엔드가 전혀 없어 화면만 예시로 떠 있는 상태.
- 매번 화면을 새로 펼쳐놓다 보니 "얘가 뭘 만들고 있는지 계속 헷갈리고 산만하다"는 공감.
- **방향**: 차라리 작은 기능 하나를 풀스펙(완전 동작)으로 만드는 게 낫다는 쪽으로 기울어짐. → 이후 범위 축소 논의로 연결.

진행 중 학습 포인트:

- Codex가 `/feature` 스킬(김지헌이 만든 것)을 자동 실행 — 기능 구현 전 feature 정의를 먼저 하게 강제하는 상위 규약. 임의 조작을 막고 협업 단위를 정리하기 위함.
- 작업 시 `feat/finz-onboarding-character` 같은 작업 브랜치를 Codex가 자동 생성.

## 5. 기술 학습 — git 브랜치/커밋/푸시 (비개발자 대상)

남덕우 대상으로 김지헌이 git 핵심 개념 설명:

- **브랜치**: 내 컴퓨터에서 만드는 작업 분기. 여러 개 생성 가능.
- **커밋(commit)**: 작업 단위를 묶는 개념 (예: 파일 10개 수정분을 하나로 묶음). 내 컴퓨터 안에서만 발생.
- **푸시(push)**: 커밋을 GitHub 서버에 올리는 행위 → 이때부터 팀원과 공유됨.
- **메인 브랜치**: 배포와 연결. main에 머지되면 100% 바로 운영 배포됨. → 그래서 **"절대 main에서 작업하지 말 것, 무슨 작업이든 반드시 브랜치를 따서 할 것"** 이라는 강력한 상위 규약을 걸어둠.
- 충돌(conflict): 같은 파일의 같은 위치를 두 명이 수정하면 발생.
- 바이브 코딩 이전엔 416줄 추가 같은 작업을 개발자가 일일이 직접 타이핑했다는 점에 모두 "혁명"이라며 감탄.

→ **액션**: 남덕우는 "git 브랜치 개념을 별도로 공부"하기로 메모. (이해하기 쉽지 않은 영역이라 다음에 다시 설명 예정)

## 6. 사용자 시나리오 검토 — 게임성 과다 우려

Codex에 "MVP 전체 사용자 시나리오"를 요청해 함께 낭독. Codex가 제시한 흐름:
취향 카드 → 캐릭터 → 친구 초대/파티 합류 → 파티 조합 해석 → 오늘의 우정주(예: 엔비디아) → 투자 레이드(각자 한 줄 포지션) → 핀즈가 대화 요약 → 다음 우정주/레이드.

검토 의견:

- **김태훈**: 우정주가 나오는 데까지는 OK. 하지만 그 이후 "각자 한 줄 의견 남기고 끝"인 투자 레이드 부분이 우리가 기대한 것과 다름. 우리가 기대한 건 ① 서로 다른 스타일의 사람들이 한 그룹으로 묶였을 때 적합한 우정주를 추천받고, ② 그 이후 AI가 최신 정보·논쟁거리를 던지며 대화를 이어가는 쪽. 지금은 **게임적 요소(캐릭터/스탯/레이드)에 비중이 과하게 쏠려** "우리가 하려던 게 이게 아닌데" 싶음.
- **남덕우**: 시나리오가 "카드 → 캐릭터 → 파티 합류 → 조합 해석"으로 매끄럽지 않음. 우리가 그리던 건 "채팅방에 들어와 캐릭터를 고르고 이야기를 진행"하는 흐름.
- 공통 인식: **조합을 해석해 '뭔가 부족하니 채워라/레이드를 해라'** 식의 게임 미션 구조는 우리 콘셉트와 안 맞음. 목적은 정답 맞히기가 아니라 **친구들이 투자 이야기를 시작하게 하는 것**.

## 7. 핵심 설계 토론 — 변수 부재와 그 해법

김지헌이 가장 근본적인 문제를 제기:

- 시나리오 1~3번(취향 카드 → 캐릭터 → 스탯)은 계정별로 **딱 한 번** 하는 단계.
- 같은 멤버가 모이면 규칙 기반으로는 **파티 조합 해석·우정주가 매번 똑같이** 나옴 → **변동성(변수)이 전혀 없음**.
- 따라서 1~6번이 매번 같은 값만 내는 구조. 이를 깰 **변수**가 필요.

변수를 만드는 수단으로 제시된 것들:

1. **채팅 누적**: 대화 기록이 쌓이면서 매번 다른 입력이 됨.
2. **AI 가이드(진행자)**: AI가 스스로 뉴스/주제/질문을 던지며 대화를 끌어감. ("탱커 스탯이 높게 나왔는데 뭘 골랐길래?" 식 질문) → 사람이 주제를 끌지 않아도 어색하지 않게.
3. **목표(goal) 개념 추가 검토**: 스탯은 '현재'를 나타내는 좋은 도구. 여기에 "이번엔 공격적으로 가보겠다" 같은 **목표치**를 대화방에서 설정 → 달성 여부에 따라 스탯이 성장하며 정체성이 형성됨. (도입 여부는 보류)

→ **AI의 역할 = 진행자 + 대화자(중재자)**. 파티의 일원처럼 질문/소재를 지속 제공하고, 대화가 쌓이는 과정 자체가 변수를 만든다.

## 8. 우정주 도출 방식 — 후보 제시 vs AI 자동 픽

우정주를 어떻게 낼지 두 방향을 비교:

- **A. 여러 후보 제시 → 사용자가 선택/교체** (실전성·실용성 강조 시 유리)
- **B. AI가 파티 조합 기반으로 하나를 자동으로 픽 ("이 파티의 우정주는 이거야")** (게임성·뽑기 느낌 강조 시 유리, 제약이 곧 미션이 됨)

논의:

- 우리 콘셉트는 실전형이 아니라 **토론/대화를 위한 것** → 제약을 주는 게임성이 더 어울림.
- 랜덤이 아니라 **우리 대화·특성을 기반으로 AI가 적절히 픽**하는 것. 폐기하려면 시도 이력이 있어야 하는 식의 제약도 가능.
- **우정주 발생 시점**: 매번 자동으로 튀어나오는 게 아니라, **대화가 무르익거나 정보·목표가 충분히 모였을 때**, 혹은 "이쯤 우정주 정해보자"는 판단 시 추천되는 형태.

→ **결정**: 일단 **AI가 파티 조합 기반으로 하나를 자동 픽**하는 방식으로 시작. ("바꾸더라도 일단 해보고 바꾸자")

## 9. MVP 범위 축소 + V0 설계 재요청

- 김지헌: MVP 범위가 너무 넓다. 레이드까지 나오면 기능 중심으로 옆으로만 늘어남. **MVP는 우정주를 뽑는 것까지**로 좁혀야 함.
- 우정주 이후 콘텐츠(매매 논의, 모의투자 등)는 "없다"가 아니라 **MVP 이후 단계로 연기**. 우정주가 나온 시점이면 이미 개인·파티 정보가 충분히 도출돼 있어 다음 콘텐츠의 기반 데이터가 됨 (가설).

**AI 기능을 V0에 포함할지** 논의:

- AI 없이 룰 기반으로만 가면 "화면밖에 못 보는" 지금 상태와 같음.
- 이 기능의 **핵심이 AI**이므로, **V0에 LLM 연결까지 포함**해 실제 동작하게 만들기로.

Codex에 재요청한 프롬프트(요지):

> "MVP 범위를 우정주를 뽑는 것까지로 한정. 이후 백엔드 작업을 포함해 완전한 MVP 기능을 동작시키기 위해 남은 작업을 알려줘. AI 연결(LLM)이 이번 MVP의 핵심이니 V0에 포함해서 설계해줘. AI가 진행자 및 대화자 역할로 대화를 이끌 수 있는 질문과 소재를 지속적으로 제공해야 한다."

Codex 응답 요지:

- 룰 기반 추천이 아니라 **파티 성향을 LLM에 넣어 오늘의 우정주를 생성**하는 제품으로 재설계.
- 남은 백엔드 작업: 실제 사용자/파티 데이터 저장, 친구 초대 링크 공유, 우정주 추천 로직.
- 저장소: 초기 **로컬 스토리지**, 공유 기능은 **Upstash KV** (트래블에서 공유하기 기능에 이미 KV 사용 — 참고).
- LLM 통합은 secondwind 트래블의 **Gemini AI** 사용 코드를 참고.

---

## 10. 토큰 비용 — 복리 증가와 파티 해산

김지헌이 운영 비용 관점 경고:

- LLM은 **상태(state) 기능이 없어** 이전 대화를 매번 전부 누적해서 보내야 함.
- 채팅이 쌓일수록 매 요청마다 보내는 대화량이 늘어 **토큰 비용이 복리로 증가**.
- 따라서 ① **파티를 언제 해산할지**를 늘 염두에 두거나, ② 파티를 유지하려면 **LLM이 인식할 대화 범주(예: 최대 N회)를 제한**해야 함. → 보류·후속 검토 항목.

## 11. 마무리 — PR 생성 및 GitHub 인증 이슈

- 작업분을 `feat/...` 브랜치에서 **PR 생성**. (main 직접 푸시 금지 규약대로) Codex가 `origin/main`을 가져와 브랜치에 반영 후 검증.
- 진행 중 작업 충돌 흔적("일본 잔재"는 개인 계정으로 시작했다가 프로젝트 형태로 분리하며 남은 것) 정리.
- 남덕우 PC에서 `gh`(GitHub CLI) **인증 미설정** 문제 발생 — `gh auth login` 프롬프트가 이 환경에서 안 떠서 푸시가 막힘. → **남덕우가 GitHub 인증 설정을 직접 해결**한 뒤 공유하기로.

## 12. 잡담 — AI 사업화·업무 적용

본 회의와 직접 관련은 없지만 기록.

- **남덕우(건설)**: AI로 건설현장 견적서 작성을 시스템화해 솔루션으로 제공하는 사업화 아이디어 구상 중. 건설사는 자체 시스템이 거의 없어 외부 제공 여지가 있음.
- **김지헌(회사 사례)**: 비마트 물류센터 ~100개의 제각각인 인력관리 방식을 인터뷰로 취합 → AI 기반으로 통합 시스템 구축한 사례. 핵심은 "AI를 쓰든 안 쓰든 **표준 양식/규칙을 먼저 만드는 작업**"이 선행돼야 한다는 것. 또한 AI를 **데이터 분석 도구**로 활용(지표 넣으면 분석)하도록 리더들에게 권장 중.

## 주요 결정사항

- **MVP 범위 축소**: 취향 카드 선택 → 캐릭터 소환 → 파티 조합 → **우정주 추천까지로 한정**. 투자 레이드 등 이후 콘텐츠는 MVP 범위에서 제외.
- **우정주 선정 방식**: 여러 후보 중 선택이 아닌, **AI가 파티 조합 기반으로 하나를 자동 픽**. (추후 변경 가능)
- **AI 기능 포함**: **V0에 LLM 연결까지 포함**해 실제 동작하는 기능으로 구현.
- **AI 역할**: 대화방의 **진행자 및 대화자(중재자)**. 주제·질문을 지속 제공.
- **저장소**: 초기 로컬 스토리지, 공유 기능은 Upstash KV.
- **LLM 통합**: secondwind 트래블의 Gemini AI 코드 참고.
- **Git 전략 재확인**: main은 배포 연결, 모든 작업은 반드시 별도 브랜치에서.

## 미결 / 보류 사항

- **파티 해산 시점**: LLM 토큰 비용(복리 증가) 관리를 위해 대화 범위 제한 또는 파티 해산 규칙 필요.
- **스탯 시스템**: 현재 스탯 외 **목표(goal) 개념** 추가 여부 검토.
- **우정주 이후 콘텐츠**: 매매 논의·모의투자 등은 MVP 이후 단계로 연기.
- **변동성 확보 방식**: 규칙 기반만으로는 매번 같은 결과 → 채팅 누적 + AI 가이드로 변수 확보 (구체 설계 미정).
