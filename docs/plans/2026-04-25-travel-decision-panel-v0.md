# travel decision panel v0

> Draft: 2026-04-25
> Context: `/travel` 의 다음 개선축을 "결정 종료율" 로 잡는다. 이 문서는 구현 전 미니 스펙이다.

## 목표

여행 계획을 생성한 사용자가 더 검색하거나 재생성하기 전에, 현재 계획을 그대로 확정할 수 있는지 빠르게 판단하게 만든다.

v0 의 성공은 "가장 완벽한 일정" 이 아니라 "이 정도면 가도 되겠다" 라는 결정을 끝내는 것이다.

## 문제

현재 `/travel` 은 일정, 지도, 이동, 비용, 공유 링크를 제공한다. 하지만 사용자가 결과를 받은 뒤 결정하지 못하게 만드는 불안은 여전히 남아 있다.

- 이 일정이 내 요청을 잘 반영했는가?
- 동선이 너무 빡세지는 않은가?
- 장소가 실제로 존재하고 찾아갈 수 있는가?
- 가격, 메뉴, 영업시간 같은 정보는 얼마나 믿어도 되는가?
- 확정한다면 지금 무엇을 확인해야 하는가?

이 불안을 줄이지 못하면 사용자는 다시 검색하거나, 다시 생성하거나, 결정을 미룬다.

## 사용자 경험

생성 결과 상단에 `결정 패널` 을 추가한다. 사용자가 긴 일정을 읽기 전에 다음 세 가지를 먼저 본다.

1. 이 일정으로 가도 되는 이유
2. 확정 전에 확인해야 할 것
3. 확정 후 할 일

패널 하단에는 `이 일정으로 확정` 버튼을 둔다. 버튼을 누르면 결과 화면이 `확정됨` 상태로 바뀌고, 공유 링크와 체크리스트를 계속 보여준다.

## 화면 구성

### 섹션 1. 이 일정으로 가도 되는 이유

사용자 요청과 생성 결과를 연결하는 짧은 근거 2~3개를 보여준다.

예시:

- 아이 동반 요청 때문에 오후 일정은 이동이 짧은 장소 위주로 묶었습니다.
- 점심과 저녁을 매일 포함했고, 식사 간격이 크게 벌어지지 않게 배치했습니다.
- 숙소 기준으로 1일차와 2일차 동선을 나누어 왕복 이동을 줄였습니다.

### 섹션 2. 확인 필요

결정을 미루게 만드는 불확실성을 숨기지 않고, 2~4개로 줄여서 보여준다.

예시:

- 영업시간과 휴무일은 방문 전 확인이 필요합니다.
- 추천 메뉴와 가격은 AI 추정입니다.
- 위치 확인 필요 표시가 있는 장소는 지도 검색 결과가 완전히 일치하지 않을 수 있습니다.
- 대중교통 이동 시간은 실제 배차에 따라 달라질 수 있습니다.

### 섹션 3. 확정 후 할 일

실제 예약 연동이 아니라 단순 체크리스트다. 사용자가 다음 행동을 잊지 않게 돕는다.

예시:

- 숙소 예약 여부 확인
- 1일차 저녁 식당 영업시간 확인
- 렌터카, 기차표, 항공권 준비 여부 확인
- 동행자에게 공유 링크 보내기

## CTA 상태

### 기본 상태

- 버튼: `이 일정으로 확정`
- 보조 문구: `확정하면 출발 전 확인할 일을 체크리스트로 볼 수 있어요.`

### 확정 후 상태

- 상태 배지: `확정됨`
- 버튼은 비활성화하거나 `확정 완료` 로 바꾼다.
- `확정 후 할 일` 체크리스트를 강조한다.
- 공유 링크 섹션을 유지한다.

## 데이터 설계

v0 는 하이브리드 방식이 적합하다.

- LLM 이 작성하는 것:
  - `decision.good_reasons`: 이 일정으로 가도 되는 이유
  - `decision.todo`: 확정 후 할 일
- 코드가 계산하거나 기존 데이터에서 파생하는 것:
  - 검증된 장소 수
  - 위치 확인 필요 장소 수
  - 예상 총 경비
  - 자동차 이동 시간의 OSRM 계산 여부
- 기존 필드에서 가져오는 것:
  - `rationale`
  - `caveats`
  - `place_warning`
  - 비용 세부 내역

후보 타입:

```ts
type DecisionSummary = {
  good_reasons: string[];
  check_before_confirming: string[];
  todo_after_confirming: string[];
};
```

`TravelPlan` 에 `decision?: DecisionSummary` 를 추가한다. 이전 공유 링크나 기존 응답을 깨지 않기 위해 optional 로 둔다.

## 저장

v0 에서는 서버 저장을 하지 않는다.

- 확정 상태는 `localStorage` 에 저장한다.
- key 는 공유 여부와 무관하게 입력값 또는 plan hash 기반으로 만든다.
- 공유 링크 생성 시 확정 상태까지 저장할지는 v0 에서는 보류한다. 공유받은 사람의 결정은 별개일 수 있다.

## 측정

초기에는 분석 도구 없이 dogfooding 으로 판단한다. 나중에 KPI 를 붙일 경우 최소 이벤트는 다음 세 개다.

- `plan_generated`
- `plan_confirmed`
- `share_created`

핵심 비율:

`plan_confirmed / plan_generated`

보조 비율:

`share_created / plan_confirmed`

## 명시적 비목표

v0 에서는 하지 않는다.

- 네이버예약, 항공권, 숙소 예약 사이트 연동
- 결제 또는 실제 예약 실행
- 계정 기반 저장
- 팀원별 체크 상태 동기화
- 무제한 자연어 수정
- 복수 대안 일정 비교

## 구현 순서

1. `TravelPlan` 타입과 프롬프트에 `decision` optional 필드를 추가한다.
2. `PlanCard` 상단에 결정 패널 UI 를 추가한다.
3. 확정 버튼과 localStorage 상태를 추가한다.
4. 기존 플랜처럼 `decision` 이 없는 경우에도 안전하게 fallback 을 보여준다.
5. 브라우저에서 생성, 확정, 새로고침, 공유 링크 화면을 QA 한다.

## 리스크

- LLM 이 좋은 점을 과장하면 신뢰가 오히려 떨어진다.
- 체크리스트가 예약 기능처럼 보이면 기대가 어긋난다.
- 확인 필요 항목이 너무 많으면 사용자가 확정하지 못한다.
- 확정 상태가 공유 링크와 섞이면 "내 확정" 과 "상대의 확정" 이 혼동될 수 있다.

## 추천

먼저 `decision` 필드를 추가하는 방식으로 간다. 프롬프트가 이미 여행자의 요청을 알고 있으므로 "왜 이 일정으로 가도 되는지" 는 LLM 이 가장 잘 쓸 수 있다. 다만 검증 상태와 비용 같은 사실성은 코드가 계산해서 패널 안에 같이 보여준다.

이렇게 하면 v0 는 작게 시작하지만, 이후 chat 기반 부분 수정과 KPI 측정으로 자연스럽게 이어질 수 있다.
