## 배경

`/travel` 의 stay-picker 는 "사용자가 자기 정보(숙소)를 LLM 에 강제 주입" 패턴을 이미 검증했다. 하지만 stay 는 1개로 한정되고, 일반 장소 (가고 싶은 카페·식당·관광지) 는 자유 요청 textarea 에 평문으로만 들어간다 → LLM 이 누락하거나 비슷한 다른 장소로 대체하는 일이 잦다.

"내 장소" 는 이 패턴을 일반화한다:
- 사용자가 평소에 모아둔 장소 컬렉션을 갖는다.
- 여행 계획 만들 때 그 컬렉션에서 N개를 골라 **"반드시 가야 할 곳"** 으로 LLM 에 전달한다.

장기적으로는 구글맵 "내 목록" 과 동등한 멘탈모델 — 그룹화된 장소 보관함, 외부 앱 공유 intent 수신, AI 자동 연결.

이 plan 은 그 중 **인증·PWA 의존성 없이 가능한 v0 골격** 까지를 다룬다.

---

## 사용자 시나리오

1. **컬렉션 만들기.** 사용자가 "제주 다음달" 리스트를 만들고 친구한테 들은 식당·카페 8곳을 텍스트로 추가한다.
2. **장소 연결.** "전부 연결" 버튼을 누르면 8곳 중 6곳이 Naver 매칭 성공 (주소·전화·카테고리·좌표 채워짐). 2곳은 "확인 필요" 상태로 남고 후보 중 하나를 수동 선택할 수 있다.
3. **여행 계획에 사용.** travel 폼에서 "내 장소에서 가져오기" → "제주 다음달" 리스트의 장소 5개 선택 → 일정 생성. LLM 이 그 5개를 동선에 맞게 배치하고, 못 넣은 항목은 rationale 에 명시.

> v0 의 자동 연결은 Naver Local Search 기반이라 **국내 장소만 매칭**. 해외 장소는 텍스트로만 저장 가능 (resolved 영구 미설정). UI 에 "해외 장소는 자동 연결 미지원 (v0)" 안내.

## Place Roles — stay vs mustVisit

travel 폼은 사용자-지정 비협상 POI 를 LLM 에 강제 주입하는 슬롯 두 개를 가짐. **둘 다 비슷하지만 의미가 다름** — 통합 안 함, 둘 다 별개 슬롯으로 유지.

| Role | 의미 | 개수 | 동작 | 모달 노출 |
|---|---|---|---|---|
| **stay** | 일정의 거점 (어디서 자고 어디로 돌아오나) | 0 또는 1 | 동선 설계의 중심점. LLM 이 일정 시작/종료를 stay 기준으로 잡음. | mustVisit 모달에서 stay 가 선택된 경우 후보 제외 (중복 의미 방지) |
| **mustVisit** | 가야 할 곳들 (시간순 미정) | 0 ~ N | LLM 이 동선·식사 시간·영업시간에 맞춰 day 에 배치 | "내 장소" 모달의 본 기능 |

**충돌 케이스**: 사용자가 stay 로 지정한 장소를 mustVisit 모달에서도 선택하려 함 → 모달에서 disabled (이미 stay 로 사용 중 안내).

**프롬프트 측면**: SYSTEM_PROMPT 의 `- stay:` 규칙과 새 `- 사용자가 user prompt 에 "필수 방문 장소" 라인을 제공한 경우:` 규칙은 독립 블록. 충돌 시 stay 우선 (거점은 1개).

**sanitize 보호 / 좌표 보존**: 두 role 다 동일 메커니즘 — exact-normalize 매칭 만 (아래 "프롬프트 변경" 섹션). 통합 함수 1개로 두 role 처리.

### 결정 사항

- 모든 메서드 `async` — localStorage 는 sync 지만 인터페이스를 미래 IO 호환으로 미리 맞춤. 호출부는 await 만 쓰면 됨.
- localStorage key: 단일 키 `secondwind:places:v1` 에 `PlacesSnapshotV1` 직렬화. 부분 갱신 시 전체 read-modify-write — v0 데이터 규모 (∼수십 리스트, ∼수백 장소) 에선 충분히 빠르고 구현 단순.
- `version: 1` 로 시작. 향후 schema 변경 시 `migrateSnapshot` 함수에서 분기.
- 컴포넌트는 인터페이스만 의존. 단일 인스턴스를 `app/(site)/travel/places/_lib/storage.ts` 에서 export.
- **SSR 안전**: `LocalStoragePlaces` 의 모든 read 메서드는 `typeof window === "undefined"` 가드로 빈 결과 반환. write 메서드는 SSR 컨텍스트에서 호출 자체가 없도록 `"use client"` 경계 강제.
- **에러 정책** (모든 write 가 try/catch 로 감싸짐):
  - `QuotaExceededError` (∼5MB 초과) → `{ ok: false, reason: "quota" }` 반환 + 토스트 "저장 공간이 가득 찼습니다. 오래된 리스트를 정리해주세요."
  - `SecurityError` (Safari private mode 등) → `{ ok: false, reason: "private" }` + 토스트 "이 브라우저에서는 저장이 제한됩니다."
  - JSON parse 실패 (corrupt root) → 빈 snapshot 으로 자동 초기화 + 토스트 "저장 데이터를 읽지 못해 초기화했습니다."
  - 무음 실패 절대 금지. 사용자가 add 했는데 안 들어가면 즉시 알림.

### 구현 노트 (Next.js App Router)

- `/travel/places` 와 `/travel/places/[listId]` 는 모두 **client component** (`"use client"`). 라우트 자체는 server-rendered shell 만 두고, 컨텐츠는 client 에서 hydrate.
- 첫 페인트는 빈 skeleton ("리스트 불러오는 중..."). `useEffect` 로 storage 에서 read 후 state 채움 → hydration mismatch 0.
- 이미 [stay-picker.tsx](app/(site)/travel/_components/stay-picker.tsx) 가 동일 패턴 (Kakao SDK 도 client-only) — 그대로 따라감.

## AI 장소 매칭

### API 경계

기존 [lib/common/services/travel-enrich.ts](lib/common/services/travel-enrich.ts) 의 모든 함수는 `assertServerEnv()` 호출 → **서버 전용**. 클라이언트에서 직접 호출 불가. 새 API endpoint 필요.

**신규 엔드포인트** (PR 3 산출물): `POST /api/places/resolve`

```ts
// 요청
type ResolveRequest = {
  name: string;              // 사용자가 입력한 장소명
  destinationHint?: string;  // list.destinationHint
};

// 응답 (union — 호출부가 한 곳에서 분기)
type ResolveResponse =
  | { kind: "auto"; place: PlaceInfo }            // 단일 고확신 후보
  | { kind: "candidates"; candidates: PlaceInfo[] } // 여러 후보 — 사용자 선택 필요
  | { kind: "none"; reason: "no-results" | "international" | "low-confidence" };
```

내부 구현은 기존 export 된 [`searchPlaceCandidates(query, hint, limit=3, cache?)`](lib/common/services/travel-enrich.ts#L291) **하나만** 사용 (verified planner 패턴과 동일):

```
candidates = searchPlaceCandidates(name, hint, 3)
  ├─ 0개 → { kind: "none", reason: "no-results" }
  ├─ 1개                       → { kind: "auto", place: candidates[0] }
  ├─ ≥2개, top1 score 압도적   → { kind: "auto", place: candidates[0] }
  │     (top1.score ≥ 0.6 AND top1.score - top2.score ≥ 0.2)
  └─ ≥2개, 동률/모호           → { kind: "candidates", candidates: top3 }
```

`searchPlaceCandidates` 는 이미 `MIN_SCORE = 0.25` filter, category mismatch reject, addressMatchesDestination 적용. 추가 결정 로직만 endpoint 안에서.

### 트리거

- **개별**: 장소 카드의 "장소 연결" 버튼. `resolved === undefined` 일 때 강조 표시. resolved 가 있어도 "다시 연결" 가능.
- **일괄**: 리스트 헤더의 "전부 연결" 버튼. `resolved` 가 없는 모든 장소를 sequential 처리.

### UX 분기

| 응답 kind | UI 동작 |
|---|---|
| `auto` | `resolved`/`resolvedAt` 즉시 채움. "✓ 자동 연결됨" 잠시 표시. |
| `candidates` | 카드 아래 후보 3개 노출 (이름·주소·카테고리). 사용자 클릭 시 그 후보를 `resolved` 로. "다른 장소" 버튼으로 해제 가능. |
| `none / no-results` | "확인 필요" 배지 + "검색어 수정" inline edit. |
| `none / international` | "v0 자동 연결 미지원 — 텍스트로만 저장됩니다" 안내. resolved 안 만듦. |

`destinationHint` 가 한국어 도시 alias 와 매칭 안 되거나 (예: "도쿄", "오사카") `searchPlaceCandidates` 가 0개 반환하면 endpoint 가 `international` 로 분기.

### 일괄 매칭 UX

- 클라이언트에서 sequential (병렬 X — Naver rate limit 보호).
- 1 RPS client-side rate limit (`setTimeout(1000)` between calls).
- 진행률 ("3/8 처리 중") + 마지막 요약 ("성공 6 · 확인 필요 2 · 실패 0").
- **취소 버튼 없음** — 일반적 N (8개 미만) × 1 RPS = 10초 미만, 취소 필요성 낮음. 진행률만으로 충분.
- **중간 실패 처리**: 한 건이 fetch 단에서 실패하면 그 장소는 "실패" 카운트로 분류 + 진행 계속. 사용자는 끝나고 요약에서 보고 개별 재시도 가능.

### 비용/quota 방어

- 기존 [lib/server/quota-store.ts](lib/server/quota-store.ts) 패턴 재사용. `/api/places/resolve` 에 동일 quota 적용 (PR 3 에 포함).
- v0 단계: 호출당 카운트만 기록, hard limit 는 추후. 일괄 매칭 spike 발생 시 limit 도입.

if (input.mustVisit && input.mustVisit.length > 0) {

```
- 사용자가 user prompt 에 "필수 방문 장소" 라인을 제공한 경우:
  - 해당 장소들을 모두 일정에 포함시킨다. 가능한 day/시간대를 동선·식사 시간·영업시간에 맞춰 배치.
  - 사용자가 제공한 장소명을 place_query 에 그대로 사용 (재해석·축약·동의어 변환 금지). 주소가 함께 제공됐으면 같은 주소의 장소를 의도한 것으로 본다.
  - 비슷한 다른 장소로 대체 금지. "근처 비슷한 카페" 식 제안 금지.
  - 거리·요일·기간 제약으로 모두 넣지 못하면 가능한 만큼만 넣고, 누락된 항목과 사유를 rationale 의 "달성 못한 부분" 에 반드시 명시.
  - 사용자 자유 요청 텍스트와 충돌하면 자유 요청을 우선하되, 그 사유를 rationale 에 명시.
```

#### sanitize 보호 (exact-normalize only)

[TRAVEL-PLAN-04](docs/plans/2026-04-26-feature-inventory.md) 의 `sanitizeGeneratedPlan` / `shouldSuppressPlaceQuery` 가 mustVisit 항목을 빈 문자열로 정리하지 않도록 보호.

- `sanitizeGeneratedPlan(plan, { protectedNames: input.mustVisit?.map(i => i.name) })` 형태로 화이트리스트 전달.
- 매칭은 **exact-normalize 만**: `s.toLowerCase().replace(/\s+/g, "")` 적용 후 정확 일치.
- **fuzzy match 사용 금지** — sanitize 는 hallucination 방어 safety filter. 유사도 휴리스틱으로 통과시키면 "메가덕불고기" vs "메가덕분식" 같은 다른 가게가 보호 받음 → 안전 필터 무력화. 이름 변형으로 보호 안 되는 건 의도된 동작 (LLM 이 사용자 이름을 그대로 써야 함, prompt 에 명시).
- 단일 mustVisit 내 동일 normalized name 이 2개 이상이면 add 시점에 dedup (UI 모달에서 차단).
- 테스트 케이스:
  - generic suppression 패턴 ("근처 카페") 과 mustVisit 가 정확 일치하면 보존.
  - "메가덕불고기" mustVisit + LLM 이 "메가덕불고기 신주쿠점" 출력 → 보호 안 됨 (의도). place_warning 으로 사용자에게 노출.

#### 매칭된 장소 좌표 보존 (exact-normalize only)

mustVisit item 에 `place` (resolved PlaceInfo) 가 있으면, 해당 일정 item 의 PlaceInfo 도 동일하게 채워서 enrich 단계 재호출 없이 통과.

- 구현 위치: `parseTravelPlan` 직후, `enrichPlan` 직전.
- 매칭 키: place_query 와 mustVisit.name 을 sanitize 보호와 동일한 exact-normalize (`s.toLowerCase().replace(/\s+/g, "")`) 정확 일치 만.
- **fuzzy 사용 금지**: 비슷한 이름에 잘못된 좌표를 강제 주입하는 게 좌표 없는 것보다 나쁨 (지도 마커가 엉뚱한 곳에 박힘).
- 매칭 성공: 해당 day item 의 `place = mustVisit.place` 그대로 복사. enrich 가 같은 query 로 다시 검색해도 cache hit 으로 비용 0.

#### mustVisit → plan 누락 검증 (post-parse diff)

LLM 이 "필수 방문 장소" 규칙을 어기고 일부 mustVisit 을 silently drop 할 수 있음 (Flash Lite 민감도). prompt 의 "rationale 에 명시" 만 신뢰하지 않고 **출력 차이를 명시적으로 계산해 사용자에게 노출**.

- 구현 위치: `parseTravelPlan` + 좌표 보존 직후.
- 알고리즘:
  ```
  expected = new Set(mustVisit.map(m => normalize(m.name)))
  actual   = new Set(plan.days.flatMap(d => d.items).map(i => normalize(i.place_query)))
  missing  = expected − actual
  ```
- `TravelPlan` 에 `mustVisitMissing?: string[]` 필드 추가 (응답에 포함, isTravelPlan validator 통과).
- PlanCard 에 명시 표시: missing 비어있지 않으면 노란 배너 — "다음 장소는 일정에 포함되지 않았습니다: X, Y. rationale 의 사유 확인."
- 테스트: mustVisit 3개 중 1개 누락 케이스 → `mustVisitMissing` 에 1개, PlanCard 배너 표시.

#### grounded planner 와의 상호작용

[`grounded`](lib/common/services/travel-grounded.ts) 모델은 사용자 prompt 키워드로 [후보 풀] 을 먼저 만들고 LLM 이 풀 안에서만 골라 → mustVisit 과 슬롯 충돌.

- **resolved mustVisit (place 있음)**: 풀에 **우선 삽입**. 검증된 데이터이므로 keyword seed 기반 후보보다 신뢰도 높음. `extractSeeds` 다음에 `prependMustVisitToPool(pool, mustVisit)` 호출.
- **unresolved mustVisit (text only)**: 자동으로 keyword seed 로 추가 → `collectPoolFromSeeds` 가 일반 검색.
- 풀 외부 처리 규칙은 그대로 — mustVisit 의 resolved 이름이 풀 안에 들어가있으니 LLM 이 그걸 place_query 로 쓰면 OK, 다른 이름 쓰면 빈 문자열로 정리.
- balanced/verified/classic 모드는 후보 풀 메커니즘 없음 → 영향 없음.

#### destinationHint timing — v0 한정

resolve 는 **장소 add 시점에 1회**. list.destinationHint 가 그때의 region context.

- 같은 장소를 다른 region 여행에서 사용 시: resolved 좌표가 list.destinationHint 기준이라 의도와 다를 수 있음. 예: "근처맛집" 리스트의 장소를 제주 여행에서 mustVisit 으로 사용 → 회사 근처 좌표 그대로.
- v0: 받아들임. UI 안내 — mustVisit 칩에 hover 시 "이 장소는 [list.name] 의 [hint] 기준으로 연결됨" 노출.
- 향후: per-place hint override 또는 trip-build 시점에 destinationHint 변경되면 자동 재해결.
