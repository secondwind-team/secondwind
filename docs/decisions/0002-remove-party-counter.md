# 0002 — travel 폼에서 인원 카운터 제거, 자유 프롬프트로 통합

> 작성: 2026-04-23 · 상태: Accepted
> 관련 결정: [0001](./0001-v0-stack-and-accepted-risks.md)

## 결정

`/travel` 폼의 구조화된 인원 입력 (`TravelParty` = 성인/청소년/어린이/영유아 카운터) 을 제거하고, 인원·동행자 정보는 자유 요청사항 텍스트로 통합한다.

- `TravelInput` 에서 `party` 필드 제거
- 백엔드 `normalizeTravelInput` 의 party 검증 로직 (`total ≥ 1`, `adults≥1 if kids>0`) 제거
- `buildTravelPrompt` 의 "인원:" 줄 제거
- `PARTY_KEYS`, `PARTY_LABELS`, `formatParty`, `partyTotal`, `TravelParty` export 삭제
- `USER_PROMPT_MAX` 300 → 1000 으로 상향 (인원·숙소·이동수단 등 더 많은 맥락을 자유 프롬프트로 담기 위해)
- 시스템 프롬프트의 "인원 구성에 맞는 장소 선정" 규칙은 유지하되, "자유 요청 텍스트에서 인원·동행자 정보를 해석" 하도록 문구 보강

## 이유

실제 dogfooding 결과 **구조화된 인원 입력이 Gemini 출력에 의미있게 반영되지 않았다.** 성인 2명 · 어린이 1명 조합과 성인 3명 조합의 플랜 차이가 미미했고, 오히려 자유 요청에 "6세 아이와 함께" 같은 맥락을 썼을 때 더 좋은 맞춤이 나왔다.

구조화 필드는 계속 유지하면:
- 백엔드 검증 (adults ≥ 1 when kids > 0 등) 을 끌고 가야 함
- 프론트에 PartyRow 같은 카운터 UI 유지비 발생
- 동일한 정보를 사용자가 카운터에서도, 자유 요청에서도 두 번 입력할 유인 발생

반면 자유 텍스트로 통합하면:
- Gemini 는 "성인 2명, 6세 아이 1명" 같은 자연어를 무리 없이 파싱
- 동행자의 **관계·특성** (양가 부모, 임산부, 거동 불편 등) 을 같은 자리에서 함께 표현 가능 — 이게 실제로 플랜 퀄리티를 더 움직임
- 프론트/백 양쪽 단순화

## 수용한 트레이드오프

- **빈 입력 리스크 증가**: 사용자가 인원을 전혀 안 적을 수 있음. → PR #2 의 가이드 양식 템플릿(라벨 "구성원:" 포함) 으로 유도하여 완화.
- **타입 안전성 약화**: 인원 수가 number 가 아닌 string 속에 섞여 있음. → 이 정보는 어차피 LLM 입력으로만 쓰이고 코드에서 숫자로 계산할 일 없음 (예산 계산 같은 로직이 생기면 재검토).

## 재검토 조건

- Gemini 가 자유 텍스트에서 인원을 누락하거나 잘못 해석하는 사례가 관찰되면 → 구조화 필드 재도입 고려
- 예산 자동 계산·할인 분기 같이 **코드에서 숫자 인원이 필요해지면** → 파싱 또는 재도입

## 후속 작업

PR #2 에서 "가이드 양식 삽입" 과 "예시 보기" 아이콘 버튼을 textarea 상단에 추가 — 구조화 UI 대신 **텍스트 템플릿 + 완성된 샘플**로 입력을 유도.
