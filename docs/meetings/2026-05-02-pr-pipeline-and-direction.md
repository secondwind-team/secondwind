## 1. 협업 워크플로우 — 개념 정리

지헌이 화이트보드 식으로 정리한 흐름:

| 단계 | 행위 | 누구 |
|---|---|---|
| 1 | **PR (Pull Request)** — 작업한 코드를 GitHub 에 올려도 되는지 요청 | 작업자 |
| 2 | **Review** — 변경 내용 검토, 코멘트 달기 (선택) | 팀 |
| 3 | **Approve** — 머지 가능하도록 승인 (필수) | 현재는 지헌만 |
| 4 | **Merge** — 메인 브랜치에 병합 → Vercel 운영 자동 배포 | 현재는 지헌만 |

핵심 개념:

- **GitHub** = 코드 저장소 (실행은 안 함)
- **Vercel** = 실제 코드가 돌아가는 서버. 메인 브랜치에 머지되면 자동으로 운영 배포가 트리거됨.
- **Preview 링크** = PR 코멘트에 자동으로 달림. 머지 전 미리 보기 (URL 무작위, 일회성).
- **Production 링크** = 머지 후 운영 URL (고정). 여기서 문제 생기면 "장애".
- **Branch** = 작업 단위. AI 도구가 자동으로 브랜치 만들고 왔다갔다 처리하므로 사용자가 직접 신경 쓸 필요는 적음.
- **`fix/` vs `feat/` 접두사** = 지헌이 룰로 강제. AI 가 작업 성격을 보고 자동 부여 (단순 텍스트 변경은 `fix` 로 인식).

### .env.local 과 보안

- `.gitignore` 에 `.env*.local` 패턴이 등록되어 있어 **API 키 등 민감 정보는 GitHub 에 절대 올라가지 않음.**
- Public repo 이므로 키가 새면 누구든 가져다 쓸 수 있어 매우 중요한 설정.
- Codex/Claude Code 가 보통 알아서 처리해주지만 개념은 알고 있을 것.

### 로컬 테스트 vs Preview

- 로컬에서 띄워서 테스트하려면 `.env.local` 에 본인 키들이 다 세팅되어야 함 (Gemini 등).
- 빌드 환경 의존성 때문에 로컬 테스트가 빡셀 수 있음 → **가급적 PR 올린 뒤 Preview 링크에서 확인하는 걸 권장.**
- 7단계는 "최소한의 서버를 띄울 수 있는 수준"까지만 깐 상태.

## 2. 시연 1 — 지헌 PC 에서 한 바퀴

샘플 작업: travel 앱 가이드 양식의 한 줄 텍스트 제거.

흐름이 잘 돌아가는 것을 확인:

1. Codex 에 "트래블 가이드 양식에서 ○○ 텍스트 빼줘" 명령
2. 자동 브랜치 생성 (`chore/travel-prompt-trim-budget-hint`)
3. PR 자동 생성 → PR 페이지에서 변경된 파일/diff 자동 요약 확인
4. PR 코멘트에 Vercel Preview 링크 생성 → 클릭해서 변경사항 미리보기
5. Merge → Vercel 이 메인 브랜치 빌드 자동 트리거 (~30초)
6. Production URL 에서 동일 텍스트 사라진 것 확인

코드 diff 를 직접 읽기보다 **PR 페이지의 자연어 요약 + Preview 링크 비교**가 더 편하다는 데 모두 동의.

## 3. 시연 2 — 태훈 PC 에서 한 바퀴 (트러블슈팅 위주)

샘플 작업: travel 앱에서 출발일 입력 시 도착일이 비어있으면 동일 일자로 자동 채우기 (이미 값이 있으면 덮어쓰지 않음).

여러 환경 이슈로 시간이 많이 소요됨. 결과적으로는 모두 해결.

### 발생한 이슈 & 해결

| # | 이슈 | 원인 추정 | 해결 |
|---|---|---|---|
| 1 | `git push` HTTPS 인증 실패 | 새로 만든 org repo 권한 문제. 태훈 계정에 write 권한이 미부여 상태였음 | 지헌이 즉시 권한 부여 |
| 2 | Codex 가 자꾸 Rube(MCP) 를 거쳐 GitHub 에 접근 시도 | Codex 세션 실행 환경에 GitHub 인증·CLI 부재 + 세션 중간 꼬임 | 새 세션으로 재시작 |
| 3 | `gh` (GitHub CLI) 미설치 | 7단계가 git 까지만 깔고 gh 는 안 깜 | `apt install gh` (sudo 비밀번호 직접 입력) |
| 4 | Codex 환경에 GitHub 인증 토큰 부재 | "코덱스라서가 아니라 코덱스 세션의 실행환경에 GitHub 인증 자체가 없는 것" | `gh auth login` (브라우저에서 OTP 코드 페어링) |
| 5 | repo 가 개인 → org 로 옮겨진 뒤 remote 정리 | 지헌이 회의 직전에 org repo 로 이전 | 지헌이 미리 준비한 명령 스니펫 그대로 Codex 에 전달 → `git remote set-url`, `git fetch`, `git checkout main`, `git pull` |
| 6 | SSH 키 미등록 | — | `ssh-keygen` → 공개키 GitHub 계정 SSH keys 페이지에 등록 |

최종적으로 push → PR 생성 → Preview 확인 → Merge 까지 성공. **공동 작업자가 새로 합류할 때마다 이 과정의 일부가 반복될 가능성이 높음** — 태훈 환경에서 한 번 태운 게 큰 자산.

### 부수적 학습

- **Codex 채팅창 = 세션.** 새 채팅을 열면 이전 컨텍스트 전부 휘발 → 작업 중 환경 설정 같은 건 문서로 남겨야 함.
- **Codex 토큰 사용량 페이지** 를 띄워두고 작업하기. 5시간 단위로 초기화. 작업 시작 시점을 5시간 단위 경계 직전에 한 번 호출해서 카운터를 미리 깎아두는 트릭.
- **Rube/MCP 우회**는 의도된 게 아님. MCP 쓰는 순간 토큰 소모량이 수십 배 증가함. GitHub 은 명령어가 명확해서 MCP 없이 `gh` CLI 로 직접 다루는 게 훨씬 효율적.

## 4. 프로젝트 방향성 — 좁히기

### 문제 의식 (지헌)

> "네가 직접 테스트할 수 없으면 이 앱은 의미가 없다."

- 현재 travel 앱은 지헌이 가본 적 없는 지역까지 다루고 있어서 검증이 사실상 불가.
- 반대로 윤영이와 함께 쓰는 할 일 관리 앱(커플렛)은 **실사용자 피드백이 즉시 들어오기 때문에 업데이트 속도가 압도적으로 빠름.**
- 결론: **실사용을 안 하는 앱은 탁상공론.** 우리 셋이 매일 손에 쥐고 쓸 주제로 범위를 좁혀야 함.

### 협업 형식 측면 (태훈)

- 우리 셋 모두 현생이 바쁨. 오프라인으로 모이기 쉽지 않고, 메시지 주고받을 시간조차 띄엄띄엄.
- **"매일 켜게 되는 앱"** 정도 강도가 아니면 피드백 루프 자체가 안 돌아감.
- 따라서 주제 선정 기준은 "관심사 교집합"보다 **"매일 쓰는 빈도"** 가 우선.

### 후보 정리

| # | 주제 | 메모 | 제약/우려 |
|---|---|---|---|
| 1 | **투자 — 관심 종목 뉴스 브리핑** | 출근 시간(예: 7시)에 맞춰 전날 미국장 등 주요 뉴스 음성 브리핑. 사용자가 관심 종목·키워드만 등록 | 주제가 너무 넓어질 수 있음 |
| 2 | **개인 자산 컨설턴트** (태훈) | 모든 자산(예적금/주식/크립토/대출) 입력 → 매크로·금리·라이프스테이지 변화에 따라 조언 | 종목 수동 입력은 한국 증권사 API 미지원이라 결국 모두 수동, 중도 포기 가능성. 민감 정보라 DB 없이 단말기 로컬만 사용하는 구조 검토 필요 |
| 3 | **사진 기반 서비스** (태훈) | 분류, 특정 인물/장소/날짜로 자동 묶기, 영상 자동 생성 등 | 비용·인프라 부담, 단말기 네이티브 의존도 ↑ |
| 4 | **음성 일기** (덕우) | 차 안에서 주저리 말하면 일자별 자동 정리 | LLM 음성 비용 ↑ + 단말기 의존 (앱 진입→녹음 버튼 누르는 마찰) |
| 5 | **음식 사진 식단 기록** | 한 장 찍으면 자동 기록 | 사진/멀티모달 영역 — 지헌 미경험 |
| 6 | **음성 뉴스 브리핑** (태훈) | 유튜브/기사 링크를 던지면 원하는 길이로 요약 음성 브리핑 | 음성 비용 ↑ |
| 7 | **날씨** | 매일 보는 빈도 자체는 높음 | 차별화 포인트 모호 |

### 지헌의 우려

- **음성·사진** 영역은 텍스트 대비 데이터 용량과 분석 비용이 크게 점프. 비용 측면에서 첫 프로젝트로는 부담.
- **단말기 장치(카메라/마이크) 의존성**이 들어오면 개발 방식이 완전히 달라짐 (지금까지 만든 흐름과 다름). 또한 "앱 진입 → 버튼 누름" 자체가 사용성 진입 장벽이 됨.
- LLM 비용도 실측이 필요. 참고로 현재 청구 내역:
  - 커플렛 (윤영이와 사용 중, 모든 액션이 LLM 호출): **약 57원**
  - travel (테스트도 거의 안 함): **약 206원**

### 합의된 가이드라인

- **MVP 기능 하나만** 정해서 그것부터 만든다. 처음부터 여러 기능을 펼치지 않는다.
- 우선 **텍스트 중심 LLM** 으로 검증 가능한 주제 선호.
- 보안 민감 정보(자산 등)를 다룰 경우 **DB 없이 단말기 로컬에만 저장**하는 구조 검토. (travel 앱이 이미 이 구조 — 참조 가능)
- 상용 서비스(investing.com 등) 벤치마킹 모드는 사이즈가 커지므로 지양. **"우리가 진짜 원하는 기능 하나"** 부터.
